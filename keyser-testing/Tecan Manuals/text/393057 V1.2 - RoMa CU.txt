Introduction 
3.1 Purpose 
This document contains detailed information about all firmware commands implemented in the firmware of the 
ROMACU 2. It also assures the upward and downward traceability between this document and the firmware 
requirements document. 
 
3.2 Scope 
This document is intended for software engineers in order to write service and setup tools or application 
software. It is not part of the end-user documentation.

3.5 Device Overview 
The DCU2 is using two identical general-purpose standalone Phillips CAN controllers called SJA1000. One of 
the interface controllers is used for the upper layer communication (HCAN) to the TeCU. The other controller is 
used for the LCAN interface to the lower level DC-Servo 2 drives.  
In the ROMA FW, one RS interface is used to control the optional BCS. 
Supplementary the DCU2 has two further RS interfaces for future expansion and several input and output lines.

3.6 Operation & Setup 
Preliminary actions have to be performed in order to operate the module. 
 
A. Installation of bootware 
B. Assure correct CAN-Bus termination (60 Ohm) 
C. Set address switch to the correct address 
D. Installation of firmware 
E. Hardware configuration 
F. Calibration 
Boot loading 
The boot itself cannot be loaded via the CAN interface. It has to be programmed on a dedicated flash 
programmer before the assembly.  
Firmware Download / Boot commands 
The firmware download is initiated with an SOF 
command if there is already a valid firmware running. The 
program now jumps into the boot mode. 
 
The boot program supports the following commands: 
 
 
‘A’ followed by the hex file name checks if the 
first three characters of the hex filename do 
correspond with the three characters <‘xxx’>. 
This protects the module from loading a firmware 
that belongs to another device. 
 
 
‘S’ initiates the firmware download process. It 
applies the program voltage for the FLASH 
memory and copies the download procedure into 
the RAM memory range. 
 
 
The command ‘E’ erases Flash application page. 
 
 
’H’ followed by a bytestream writes directly into 
the FLASH memory. A ‘EOF’ string indicates the 
end of the Hexfile. A redundancy checksum is 
calculated and compared with a desired value. 
The program jumps into the firmware page and 
executes the downloaded code. In case of a 
deviation between checksums the program 
remains in the boot mode. 
 
 
The command ‘L’ reads the type of equipment. 
 
 
‘N’ reads the number of nodes 
 
 
‘X’ exits boot mode.

Round Corners in Roma can be achieved by starting the second motion prematurely. 
The Target Window defines the position of the first axis, when the second axis has to be started. 
 
The agminated shape (radius and symmetry) of the entire motion depends of the predefined target window, 
speed and acceleration settings.

3.9 Barcode Scanner 
The Barcode Scanner is annexed to the FW by a serial communication interface. This FW provides a 
communication pipeline straight to the BCS. It is possible to work with the BCS completely using this pipeline, 
but there are also commands implemented to handle more abstract functionalities like scanning and polling a 
barcode or keep turned on the beamer. With these special commands the required timeouts and buffer handling 
are automatically managed. 
 
 
The Barcode Scanner works with three different operation modes: 
 
 
 
 
Set Mode 
 
In Set Mode all required settings can be 
set (e.g. Barcode Type or Length of 
Barcodes). Use the TRB command to 
configure the settings. 
Scan Mode 
 
In this mode a Barcode can be 
scanned or verificated.  Scan Mode is 
the default mode. 
Beam On Mode 
 
In Beam On Mode the Beamer is switched 
on permanently. The position of the BCS 
can be adjusted. In beam on mode the test 
button on the BCS should not be used.

4.6 Action Commands (ROMA Specific Functionality) 
PIA 
Position Initialization X/Y/Z/R/G – Axis: 
PIA Selector 
This is the command, to initialize the ROMA X/Y/Z/Rotator and Gripper-axis. No movement command for 
the ROMA will be accepted before the PIA or an axis specific initialization command is executed. When 
the PIA command is entered, all drives try to move to their initialization positions with a default init speed. 
With the selector it is possible to disable the X-axis. This is useful after a pre initialization with PIB 
command. When the init position of each axis has been detected, they move to an initial offset position. It 
is possible that the ROMA can't find a free position to initialize all drives, in this case the ROMA must be 
moved to a free location manually. The initialization procedure depends on the selected hardware defined 
by the SHW command.

PIB 
Position Pre Initialization: 
PIB 
The PIB command is used to perform a pre initialization of the arm. This must be to prevent a collision 
with an obstacle in the rear part of the instrument while initialization. After performing the PIB, normally a 
PIA without X-axis will be performed. The goal of this command is to bring the Y-axis to the front of the 
instrument. After doing a PIB, all axis have a not initialized status. The initialization procedure depends 
on the selected hardware defined by the SHW command.
